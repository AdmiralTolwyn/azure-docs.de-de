Änderungen, die über das Azure-Portal, SMB oder REST an der Azure-Dateifreigabe vorgenommen wurden, werden nicht sofort erkannt und wie Änderungen am Serverendpunkt repliziert. Änderungen werden nicht sofort repliziert, da Azure Files noch über keine Änderungsmitteilung (Journaling) verfügt, sodass es keine Möglichkeit gibt, eine Synchronisierungssitzung automatisch zu starten, sobald Dateien geändert werden. Unter Windows Server verwendet Azure File Sync das [Windows-USN-Journaling](https://msdn.microsoft.com/library/windows/desktop/aa363798.aspx), um automatisch eine Synchronisierungssitzung zu starten, wenn sich Dateien ändern.

Um Änderungen an der Azure-Dateifreigabe zu erkennen, verfügt Azure File Sync über einen geplanten Auftrag namens „Änderungserkennungsauftrag“, der jede Datei in der Dateifreigabe aufzählt und sie mit der Synchronisierungsversion dieser Datei vergleicht. Wenn der Änderungserkennungsauftrag feststellt, dass sich Dateien geändert haben, startet Azure File Sync eine Synchronisierungssitzung. Der Änderungserkennungsauftrag wird alle 24 Stunden ausgelöst. Da der Änderungserkennungsauftrag darauf abzielt, jede Datei in der Dateifreigabe von Azure aufzuzählen, ist bei großen Namespaces mehr Zeit erforderlich als bei kleineren. Das bedeutet, dass es für diese Namespaces länger als einmal alle 24 Stunden dauern kann, um festzustellen, welche Dateien sich geändert haben. 

Langfristig möchten wir eine Änderungserkennung in Azure Files ähnlich wie USN unter Windows Server hinzufügen.
